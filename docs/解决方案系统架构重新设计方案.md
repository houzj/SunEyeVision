# 解决方案系统架构重新设计方案

## 一、问题分析

### 1.1 当前架构的问题

#### 问题1：元数据重复
- **SolutionMetadata**（SolutionMetadataReader.cs）：临时数据结构，用于快速读取
- **SolutionInfo**（RuntimeConfig.cs）：持久化数据结构，用于最近使用记录
- **重复属性**：Id, Name, Description, Version, FilePath, CreatedTime, ModifiedTime, LastAccessTime
- **维护成本高**：修改时需要同步两个类

#### 问题2：命名不清晰
- **RuntimeConfig**：名字过于通用，无法反映实际功能
- **实际职责**：用户会话状态管理（最近使用、偏好设置）
- **视觉软件行业标准**：应使用更明确的命名

#### 问题3：职责混乱
- **SolutionManager** 承担过多职责：
  - 解决方案文件操作（加载/保存/删除）
  - 当前解决方案管理
  - 工作流管理
  - 全局变量管理
  - 设备管理
  - 通讯配置管理
  - 用户配置管理（KnownSolutions）
  - 自动加载逻辑

#### 问题4：不符合元数据驱动架构
- 没有独立的元数据管理系统
- 元数据仅作为临时工具（SolutionMetadataReader）
- 缺少元数据注册、查询、缓存机制
- 直接操作 Solution 对象，性能和内存占用高

### 1.2 影响范围

**涉及的文件**：
- src/Workflow/SolutionMetadataReader.cs（110行）
- src/Workflow/RuntimeConfig.cs（242行）
- src/Workflow/SolutionManager.cs（808行）
- src/Workflow/Solution.cs（509行）
- UI层的Solution相关ViewModel和Dialog

**影响的功能**：
- 解决方案列表显示（依赖元数据）
- 最近使用记录
- 解决方案快速加载
- 解决方案创建/打开/保存/删除

**优先级**：
- 🔴 Critical（用户体验核心功能）

### 1.3 约束条件

- 需要保持向后兼容（UI层已大量使用）
- 不能影响现有功能
- 必须遵循项目规范（命名、日志、属性通知）
- 必须符合视觉软件行业标准

## 二、新架构设计

### 2.1 设计目标

1. **统一元数据管理**：消除重复，建立单一元数据模型
2. **清晰职责分离**：Registry（注册）、Cache（缓存）、Repository（仓库）、Settings（设置）、Manager（管理）
3. **懒加载优化**：通过元数据避免加载完整Solution对象
4. **符合行业规范**：命名和结构符合视觉软件行业习惯
5. **性能优化**：元数据缓存、LRU策略、批量操作

### 2.2 核心组件

#### 2.2.1 SolutionMetadata（统一元数据类）

**职责**：解决方案的轻量级元数据表示

**特性**：
- 继承 `ObservableObject`（属性变化通知）
- 支持 JSON 序列化（用于 Settings）
- 包含所有必要的元数据字段

**字段**：
```csharp
- Id: string                    // 解决方案ID
- Name: string                  // 解决方案名称
- Description: string           // 解决方案描述
- Version: string               // 解决方案版本
- FilePath: string              // 文件路径
- DirectoryPath: string         // 目录路径
- CreatedTime: DateTime         // 创建时间
- ModifiedTime: DateTime        // 修改时间
- LastAccessTime: DateTime      // 最后访问时间
- WorkflowCount: int            // 工作流数量
- GlobalVariableCount: int      // 全局变量数量
```

**实现文件**：
- `src/Workflow/SolutionMetadata.cs`

#### 2.2.2 SolutionRegistry（元数据注册表）

**职责**：解决方案元数据的注册和查询

**特性**：
- 内存注册表（快速查找）
- 线程安全访问
- 支持批量注册和查询

**核心方法**：
```csharp
- Register(metadata): void                    // 注册元数据
- Unregister(solutionId): void                // 注销元数据
- Get(solutionId): SolutionMetadata?          // 查询指定解决方案
- GetAll(): List<SolutionMetadata>            // 查询所有
- Query(predicate): List<SolutionMetadata>    // 条件查询
- Contains(solutionId): bool                   // 检查是否存在
```

**实现文件**：
- `src/Workflow/SolutionRegistry.cs`

#### 2.2.3 SolutionCache（元数据缓存）

**职责**：元数据缓存管理，提高性能

**特性**：
- LRU（最近最少使用）淘汰策略
- 过期时间控制
- 线程安全
- 可配置缓存大小和过期时间

**核心方法**：
```csharp
- Get(key): SolutionMetadata?          // 获取缓存
- Set(key, metadata): void             // 设置缓存
- Invalidate(key): void                // 失效指定缓存
- InvalidateAll(): void                // 清空所有缓存
- GetSize(): int                       // 获取缓存大小
```

**配置**：
```csharp
- MaxSize: int (默认 100)              // 最大缓存数量
- ExpirationTime: TimeSpan (默认 30分钟) // 过期时间
```

**实现文件**：
- `src/Workflow/SolutionCache.cs`

#### 2.2.4 SolutionRepository（文件仓库）

**职责**：解决方案文件的加载、保存、删除操作

**特性**：
- 文件系统操作
- JSON 序列化/反序列化
- 批量操作支持
- 错误处理和日志记录

**核心方法**：
```csharp
- Load(filePath): Solution?                    // 加载完整Solution
- LoadMetadata(filePath): SolutionMetadata?   // 仅加载元数据
- Save(solution, filePath): bool               // 保存Solution
- Delete(filePath): bool                       // 删除文件
- Exists(filePath): bool                      // 检查文件是否存在
- LoadMetadataBatch(filePaths): List<...>     // 批量加载元数据
```

**实现文件**：
- `src/Workflow/SolutionRepository.cs`

#### 2.2.5 SolutionSettings（用户设置）

**职责**：用户偏好设置和最近使用记录

**特性**：
- 持久化到 JSON 文件
- 继承 `ObservableObject`（支持 UI 绑定）
- 线程安全集合

**字段**：
```csharp
- CurrentSolutionId: string              // 当前解决方案ID
- RecentSolutions: ObservableCollection<SolutionMetadata>  // 最近使用的解决方案（按访问时间排序，最多20个）
- UserPreferences: Dictionary<string, object>              // 用户偏好设置
```

**核心方法**：
```csharp
- AddRecentSolution(metadata): void      // 添加到最近使用
- RemoveRecentSolution(solutionId): void // 从最近使用中移除
- ClearRecentSolutions(): void           // 清空最近使用
- GetPreference<T>(key, default): T?    // 获取偏好设置
- SetPreference(key, value): void       // 设置偏好设置
- Load(filePath): void                  // 从文件加载
- Save(filePath): void                  // 保存到文件
```

**实现文件**：
- `src/Workflow/SolutionSettings.cs`

#### 2.2.6 SolutionManager（高层管理器）

**职责**：高层协调，整合所有组件

**特性**：
- 组合所有组件
- 提供统一的高层API
- 触发事件通知UI层
- 管理当前解决方案

**依赖**：
```csharp
- SolutionRegistry _registry          // 元数据注册表
- SolutionCache _cache                 // 元数据缓存
- SolutionRepository _repository       // 文件仓库
- SolutionSettings _settings           // 用户设置
- ILogger _logger                      // 日志系统
```

**核心方法**：
```csharp
// 解决方案生命周期
- CreateNew(name, description): Solution           // 创建新解决方案
- OpenSolution(filePath): Solution?               // 打开解决方案（懒加载）
- CloseSolution(): void                           // 关闭当前解决方案
- SaveSolution(filePath?): bool                   // 保存当前解决方案
- DeleteSolution(solutionId): void                // 删除解决方案

// 元数据管理
- GetMetadata(solutionId): SolutionMetadata?      // 获取元数据（优先从缓存）
- GetAllMetadata(): List<SolutionMetadata>        // 获取所有元数据
- RefreshMetadata(): void                         // 刷新元数据（扫描文件系统）
- ScanDirectory(directoryPath): void              // 扫描目录并注册元数据

// 最近使用管理
- AddToRecent(solutionId): void                   // 添加到最近使用
- GetRecentSolutions(): List<SolutionMetadata>    // 获取最近使用列表

// 批量操作
- LoadMetadataBatch(filePaths): List<...>         // 批量加载元数据
```

**事件**：
```csharp
- SolutionOpened: EventHandler<Solution>         // 解决方案打开
- SolutionSaved: EventHandler<Solution>          // 解决方案保存
- SolutionClosed: EventHandler                     // 解决方案关闭
- MetadataChanged: EventHandler                   // 元数据变更
```

**实现文件**：
- `src/Workflow/SolutionManager.cs`（重构）

## 四、风险评估

### 4.1 技术风险

#### 风险1：线程安全问题
- **描述**：多线程访问元数据注册表和缓存可能导致数据竞争
- **缓解措施**：
  - 使用 `ReaderWriterLockSlim` 保护注册表
  - 使用 `ConcurrentDictionary` 实现缓存
  - 提供线程安全的批量操作方法

#### 风险2：内存占用
- **描述**：大量元数据可能占用过多内存
- **缓解措施**：
  - LRU 缓存自动淘汰
  - 设置缓存大小限制（默认 100）
  - 设置过期时间（默认 30 分钟）

#### 风险3：性能问题
- **描述**：元数据扫描可能较慢
- **缓解措施**：
  - 使用批量操作
  - 使用并行处理（Parallel.ForEach）
  - 缓存元数据避免重复扫描

### 4.2 兼容性风险

#### 风险1：UI 层大量使用旧 API
- **描述**：UI 层可能大量使用 `RuntimeConfig` 和 `SolutionInfo`
- **缓解措施**：
  - 保留 `SolutionInfo` 类（标记为 Obsolete，提供自动映射）
  - 保留 `RuntimeConfig` 类（标记为 Obsolete，内部委托给新组件）
  - 提供迁移指南

#### 风险2：配置文件格式变化
- **描述**：`SolutionSettings` 的 JSON 格式可能与 `RuntimeConfig` 不同
- **缓解措施**：
  - 实现兼容性转换逻辑
  - 自动迁移旧配置文件

### 4.3 回退方案

如遇重大问题，可回退到当前架构：
- 保留旧代码（标记为 Legacy）
- 新代码作为实验性功能
- 逐步迁移 UI 层

## 六、对比分析

### 6.1 旧架构 vs 新架构

| 维度 | 旧架构 | 新架构 |

## 七、迁移指南

### 7.1 API 变更

#### 7.1.1 SolutionInfo → SolutionMetadata

**旧代码**：
```csharp
var info = new SolutionInfo
{
    SolutionId = "sol_001",
    Name = "Solution A",
    FilePath = "C:/Solutions/A.solution"
};
RuntimeConfig.AddOrUpdateSolution(info);
```

**新代码**：
```csharp
var metadata = new SolutionMetadata
{
    Id = "sol_001",
    Name = "Solution A",
    FilePath = "C:/Solutions/A.solution"
};
SolutionManager.AddToRecent(metadata.Id);
```

#### 7.1.2 RuntimeConfig → SolutionSettings

**旧代码**：
```csharp
var config = SolutionManager.RuntimeConfig;
config.SetPreference("key", value);
var value = config.GetPreference<string>("key");
```

**新代码**：
```csharp
var settings = SolutionManager.Settings;
settings.SetPreference("key", value);
var value = settings.GetPreference<string>("key");
```

#### 7.1.3 SolutionMetadataReader → SolutionRepository

**旧代码**：
```csharp
var metadata = SolutionMetadataReader.ReadMetadata(filePath);
```

**新代码**：
```csharp
var metadata = SolutionRepository.LoadMetadata(filePath);
// 或使用缓存（推荐）
var metadata = SolutionManager.GetMetadata(solutionId);
```
