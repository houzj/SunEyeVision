## 解决方案重命名功能完整实施

### 问题背景
用户反馈：编辑解决方案名称后，重启软件发现重命名未生效。

**根本原因**：
- ❌ 物理文件未重命名（`旧名称.solution` 仍然存在）
- ❌ FilePath 未更新（仍指向旧文件）
- ❌ KnownSolutions 设置未同步

### 实施方案

#### 1. SolutionRepository.cs - 添加 `Move` 方法
**文件**: `src/Workflow/SolutionRepository.cs`

**新增方法**: `Move(string sourcePath, string destPath, bool overwrite = false)`

**功能**：
- 移动/重命名解决方案文件
- 检查源文件和目标路径有效性
- 支持覆盖模式
- 自动创建目标目录
- 完整的错误处理和日志记录

#### 2. SolutionManager.cs - 添加 `RenameSolutionFile` 方法
**文件**: `src/Workflow/SolutionManager.cs`

**新增方法**: `RenameSolutionFile(string solutionId, string newName)`

**完整流程**：
1. 参数验证
2. 获取元数据
3. 构建新文件路径
4. 检查目标文件冲突
5. 更新 Solution 对象（当前已加载或临时加载）
6. 重命名物理文件
7. 保存 Solution 到新文件
8. 更新元数据
9. 更新注册表和缓存
10. 更新 KnownSolutions 设置
11. 触发 MetadataChanged 事件
12. **回滚机制**：如果保存失败，将文件移回原位置

#### 3. SolutionConfigurationDialogViewModel.cs - 修改 `EditSolution` 方法
**文件**: `src/UI/ViewModels/SolutionConfigurationDialogViewModel.cs`

**修改内容**：
- **名称变化时**: 调用 `SolutionManager.RenameSolutionFile`（包含文件重命名）
- **仅描述变化时**: 直接加载 Solution 更新描述（不需要重命名文件）
- 从注册表获取最新元数据并更新 UI 列表
- 错误处理: 失败时显示明确的错误提示

### 设计亮点

1. **完整的回滚机制**: 如果保存失败，自动将文件移回原位置
2. **智能加载策略**: 如果 Solution 已加载则直接使用，否则临时加载
3. **事件驱动**: 触发 MetadataChanged 事件，通知 UI 层更新
4. **日志记录完整**: 每个步骤都有详细的日志记录
5. **符合规范**:
   - 命名规范（rule-002）：PascalCase 方法命名
   - 日志规范（rule-003）：使用 VisionLogger 和适当的日志级别
   - 方案设计规范（rule-004）：完整的流程和错误处理
