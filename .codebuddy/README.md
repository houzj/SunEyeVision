## 📋 规则检查清单

### 什么是规则检查清单？

规则检查清单是一个**强制执行的检查项列表**，确保所有方案设计和代码修改都遵循项目规则。

### 何时使用？

- ✅ 方案生成前
- ✅ 代码修改前
- ✅ 代码审查前

### 如何使用？

1. **打开清单**: 阅读 `rules/templates/rules-checklist.md`
2. **选择规则**: 根据任务类型，找到相关的规则
3. **逐项检查**: 对照每个规则的检查项，确认是否满足
4. **标记状态**: 使用 `[ ]` 标记未完成，`[x]` 标记已完成
5. **记录例外**: 如果有特殊情况，记录原因和批准人

### 关键规则（必须检查）

| 规则ID | 规则名称 | 优先级 | 使用场景 |
|-------|---------|-------|---------|
| rule-001 | 属性更改通知统一规范 | 🔴 Critical | 涉及属性通知 |
| rule-002 | 命名规范 | 🟠 High | 创建类、方法、变量 |
| rule-003 | 日志系统使用规范 | 🟠 High | **⚠️ 所有场景都必须检查** |
| rule-008 | 原型设计期代码纯净原则 | 🔴 Critical | 重构、删除代码 |
| rule-004 | 方案设计要求 | 🟠 High | 设计方案 |

### 示例

```markdown
## 规则检查

### rule-003: 日志系统使用规范

检查项:
- [ ] ❌ 没有使用 System.Diagnostics.Debug.WriteLine()
- [ ] ❌ 没有使用 System.Diagnostics.Trace.WriteLine()
- [ ] ❌ 没有使用 Console.WriteLine()
- [ ] ❌ 没有直接操作 UI 日志显示（如 AddLogToUI）
- [x] ✅ ViewModel 层使用 LogInfo/LogError/LogSuccess/LogWarning

结果: ✅ 通过
```

## 🧠 方案设计思维模板

### 什么是思维模板？

思维模板是一个**强制执行的思考流程**，帮助 AI 助手在生成方案时系统地考虑规则。

### 何时使用？

- ✅ 生成方案前
- ✅ 设计解决方案时

### 如何使用？

按照以下步骤思考和验证：

```
1. 理解用户问题
   ↓
2. 规则检索
   ↓
3. 违反规则分析
   ↓
4. 设计解决方案
   ↓
5. 自我审查
   ↓
6. 提交方案
```

### 步骤详解

#### 第一步：理解用户问题

**问题识别清单**:
- [ ] 用户遇到了什么问题？
- [ ] 涉及哪些文件和代码？
- [ ] 问题发生在哪个层级？
- [ ] 用户期望什么结果？

**示例**:

```markdown
## 🎯 问题分析

### 用户意图
修复 MainWindow.xaml.cs 中的日志方法使用问题

### 涉及的代码
- 文件: src/UI/Views/Windows/MainWindow.xaml.cs
- 方法: AddLogToUI (line 107-143)
- 调用: AddLogToUI("❌ 没有选中工作流标签页") (line 1891)

### 问题层级
UI 层

### 期望结果
使用正确的日志方法记录日志
```

#### 第二步：规则检索

**规则检索流程**:
1. 根据问题类型搜索相关规则
2. 读取规则内容
3. 记录规则要求

**常见问题类型与规则映射**:

| 问题类型 | 关键词 | 相关规则 |
|---------|--------|---------|
| 日志输出 | "日志" | rule-003 |
| 属性通知 | "属性"、"通知" | rule-001 |
| 命名 | "命名" | rule-002 |
| 方案设计 | "方案"、"设计" | rule-004, rule-010 |
| 原型设计 | "原型"、"重构" | rule-008 |

**示例**:

```markdown
## 🔍 规则检索

### 问题类型
日志输出

### 相关规则
rule-003: 日志系统使用规范

### 规则要求
- ❌ 禁止：Debug.WriteLine, Trace.WriteLine, Console.WriteLine
- ❌ 禁止：直接操作 UI 日志显示（如 AddLogToUI）
- ✅ 正确：ViewModel.LogInfo/LogError/LogSuccess/LogWarning
- ✅ 正确：Service._logger.Log()
```

#### 第三步：违反规则分析

**分析框架**:
1. 识别违反的规则
2. 分析违反的影响
3. 确定正确的做法

**示例**:

```markdown
## 🚨 违反规则分析

### 违反的规则
rule-003: 日志系统使用规范

### 具体问题
使用了 AddLogToUI 方法，绕过了项目的日志系统

### 影响分析
- 日志没有通过统一的日志系统输出
- 其他组件无法正确处理这些日志
- 日志级别不明确（Error vs Warning）

### 正确的做法
使用 ViewModel 的日志方法：
```csharp
_viewModel.LogWarning("没有选中工作流标签页");
```
```

#### 第四步：设计解决方案

**方案设计框架**:
1. 方案概述
2. 涉及的规则
3. 技术方案
4. 实施步骤
5. 验证清单
6. 风险控制

**示例**:

```markdown
## 💡 解决方案

### 涉及的规则
- rule-003: 日志系统使用规范
- rule-008: 原型设计期代码纯净原则

### 技术方案

#### 1. 删除 AddLogToUI 方法
**文件**: src/UI/Views/Windows/MainWindow.xaml.cs  
**位置**: Lines 107-143  
**理由**: 遵循 rule-008

#### 2. 修改日志调用
**修改前**: AddLogToUI("❌ 没有选中工作流标签页");  
**修改后**: _viewModel.LogWarning("没有选中工作流标签页");  
**理由**: 遵循 rule-003

### 实施步骤
1. 删除 AddLogToUI 方法
2. 修改日志调用
3. 验证编译通过

### 验证清单
- [x] 遵循 rule-003
- [x] 遵循 rule-008
```

#### 第五步：自我审查

**自我审查清单**:
1. 规则遵守审查
2. 方案质量审查
3. 代码质量审查
4. 验证清单审查

**示例**:

```markdown
## ✅ 自我审查

### 规则遵守审查
- [x] 对照了所有相关规则
- [x] 明确标注了遵循的规则
- [x] 没有违反任何规则

### 方案质量审查
- [x] 方案考虑了整体架构
- [x] 复用了现有基础设施
- [x] 遵循了原型设计期原则

### 代码质量审查
- [x] 命名符合规范
- [x] 日志使用正确
- [x] 删除了旧代码

### 验证清单审查
- [x] 验证清单完整
- [x] 所有检查项通过
```

## 🎯 最佳实践

### 1. 强制执行

**规则**: 在生成任何方案前，必须执行规则检查和思维模板

**如何做**:
- 第一步：理解问题
- 第二步：检索规则
- 第三步：分析违反
- 第四步：设计方案
- 第五步：自我审查

**示例**:

```markdown
# ❌ 错误：直接给出方案

用户: "修复日志问题"

AI: "直接删除 AddLogToUI 方法，改用 LogError"

# ✅ 正确：遵循思维模板

AI: "让我先理解问题，然后检查相关规则..."

1. 理解问题：使用了 AddLogToUI 方法
2. 规则检索：rule-003 日志系统使用规范
3. 违反分析：违反了 rule-003
4. 方案设计：使用 ViewModel.LogWarning
5. 自我审查：✅ 通过
```

### 2. 明确标注规则

**规则**: 在方案中必须明确标注遵循的规则

**如何做**:
```markdown
## 涉及的规则
- rule-003: 日志系统使用规范
- rule-008: 原型设计期代码纯净原则
```

### 3. 使用验证清单

**规则**: 方案完成后，必须对照验证清单逐项检查

**如何做**:
```markdown
## 验证清单
- [ ] 遵循 rule-001
- [ ] 遵循 rule-002
- [ ] 遵循 rule-003 ✅
- [ ] 遵循 rule-008 ✅
```

### 4. 记录例外

**规则**: 如果有特殊情况，必须记录原因和批准人

**如何做**:
```markdown
## 例外情况

### 违反 rule-001 的原因
**违反内容**: WorkflowNode 直接实现了 INotifyPropertyChanged

**原因**: 
1. 需要支持属性变更批处理
2. 需要扩展事件

**批准人**: 技术负责人
**批准日期**: 2026-03-24
```

## 💡 常见问题

### Q1: 为什么需要规则检查清单？

**A**: 规则检查清单确保所有方案设计和代码修改都遵循项目规则，避免遗漏重要检查项，提高代码质量和一致性。

### Q2: 如何快速找到相关规则？

**A**: 使用 `rules-index.md` 中的快速查找表格，根据问题类型找到对应的规则ID和规则名称。

### Q3: 如果违反了规则怎么办？

**A**: 
1. 识别违反的规则
2. 分析违反的影响
3. 设计解决方案（遵循规则）
4. 验证修复后不再违反

### Q4: 可以违反规则吗？

**A**: 可以，但必须：
1. 有充分的理由
2. 记录原因和批准人
3. 经过技术负责人的批准

### Q5: 如何确保方案质量？

**A**: 
1. 遵循思维模板的五个步骤
2. 使用规则检查清单逐项验证
3. 执行自我审查
4. 明确标注遵循的规则
