## 产品概述

修复解决方案仓库的元数据加载问题，确保架构变更后的代码一致性

## 核心功能

- 修复 SolutionRepository.LoadMetadata() 方法：从文件名推断 Name，从文件系统读取时间信息
- 修复 SolutionManager.LoadMetadataBatch() 方法：从 KnownSolutions 补充详细元数据
- 确保所有使用 LoadMetadata 的场景正常工作
- 完成基础功能测试验证

## 技术栈

- 语言: C# (.NET 8)
- JSON处理: System.Text.Json
- 日志系统: VisionLogger (项目自定义)

## 实现方案

### 架构变更背景

Solution 类已将元数据分离到 SolutionMetadata，JSON 文件中不再包含 Name、Description、CreatedTime、ModifiedTime 等属性。

### 核心修复策略

**1. SolutionRepository.LoadMetadata() 修复**

- Name: 从文件名推断，使用 `Path.GetFileNameWithoutExtension(filePath)`
- Description: 设为空字符串（KnownSolutions 为权威来源）
- 时间信息: 从文件系统读取 `FileInfo.CreationTime` 和 `FileInfo.LastWriteTime`
- Id/Version: 保持从 JSON 读取（仍在 Solution 中）

**2. SolutionManager.LoadMetadataBatch() 修复**

- 加载元数据后，遍历匹配 KnownSolutions 中的记录
- 使用 KnownSolutions 中的 Name、Description、Time 覆盖推断值
- 记录详细日志，便于调试

### 数据流

```
JSON文件 → 读取 Id、Version、统计数据
文件名 → 推断 Name
文件系统 → 推断 CreatedTime、ModifiedTime
KnownSolutions → 补充 Name、Description、Time（如存在）
```

## 目录结构

```
src/Workflow/
├── SolutionRepository.cs      # [MODIFY] LoadMetadata 方法，从文件名推断 Name
└── SolutionManager.cs         # [MODIFY] LoadMetadataBatch 方法，补充 KnownSolutions 同步
```

## 实施注意事项

- 保持向后兼容：现有的 KnownSolutions 数据仍然有效
- 性能考虑：KnownSolutions 通常 <100 条，遍历影响可忽略
- 日志规范：使用 VisionLogger，记录元数据来源
