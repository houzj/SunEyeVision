## 01-编码规范检查

### rule-001: 属性更改通知统一规范 (🔴 Critical)

- [ ] 所有需要属性通知的类都继承了 ObservableObject 或其派生类
- [ ] 属性使用 SetProperty 方法而不是手动实现
- [ ] 没有重复的 INotifyPropertyChanged 实现
- [ ] 如果直接实现了 INotifyPropertyChanged，有详细注释说明原因
- [ ] 正确使用 displayName 参数记录日志（对用户可见的属性）
- [ ] 对内部属性不记录日志（不带 displayName 参数）

**违规示例**:
```
❌ 直接实现 INotifyPropertyChanged
❌ 手动实现属性设置
❌ 重复实现 SetProperty
```

**检查文件**:
- [ ] ViewModel 层文件
- [ ] Model 层文件
- [ ] Plugin.SDK 层文件

---

### rule-002: 命名规范 (🟠 High)

- [ ] 变量使用小驼峰：userName, imageWidth
- [ ] 私有字段前缀下划线：_privateField
- [ ] 常量使用大写下划线：MAX_RETRY_COUNT
- [ ] 接口前缀 I：IPluginManager
- [ ] 属性使用 PascalCase：UserName
- [ ] 方法使用 PascalCase：ProcessImage()
- [ ] 类名使用名词：WorkflowEngine
- [ ] 避免缩写：使用 ImageProcessor 而非 ImgProc
- [ ] 布尔值前缀 Is/Has/Can：IsEnabled, HasChildren
- [ ] 事件使用过去式：PropertyChanged, Loaded
- [ ] 枚举使用单数：LogLevel, NodeStatus
- [ ] 命名空间使用 PascalCase：SunEyeVision.UI.ViewModels

**违规示例**:
```
❌ 使用缩写：ImgProc, CfgMgr
❌ 命名不符合视觉软件风格
❌ 使用互联网风格命名
```

**检查文件**:
- [ ] 所有 C# 源文件
- [ ] 特别关注新增和修改的类、方法、属性

---

### rule-003: 日志系统使用规范 (🟠 High)

- [ ] 没有使用 `System.Diagnostics.Debug.WriteLine()`
- [ ] 没有使用 `System.Diagnostics.Trace.WriteLine()`
- [ ] 没有使用 `Console.WriteLine()`
- [ ] ViewModel 层使用 `LogInfo()`, `LogSuccess()`, `LogWarning()`, `LogError()`
- [ ] Service 层使用 `_logger.Log()`
- [ ] UI 层通过控件日志接口记录日志
- [ ] 日志级别使用恰当（Info/Success/Warning/Error/Fatal）
- [ ] 日志信息包含足够的上下文信息
- [ ] 异常信息包含详细的错误描述和堆栈信息

**违规示例**:
```
❌ System.Diagnostics.Debug.WriteLine("调试信息")
❌ System.Console.WriteLine("调试信息")
❌ 日志信息不明确
❌ 日志级别使用不当
```

**检查文件**:
- [ ] ViewModel 层文件
- [ ] Service 层文件
- [ ] UI 层文件

## 审查流程

### 1. 快速浏览（5-10分钟）
- 了解改动的主要目的和范围
- 快速检查代码结构和大致内容

### 2. 详细检查（20-30分钟）
- 按照检查清单逐项检查
- 关注 Critical 和 High 优先级的规则
- 记录发现的问题

### 3. 反馈和沟通（10-15分钟）
- 与开发者讨论发现的问题
- 提供具体的改进建议
- 确认问题的严重性和优先级

### 4. 审查结论（5分钟）
- 填写审查结论表格
- 给出通过/不通过的建议
- 提供总体的反馈和评价

---

## 审查者注意事项

1. **时间投入**：
   - 小改动（<100行）：20-30分钟
   - 中等改动（100-500行）：30-60分钟
   - 大改动（>500行）：60-120分钟

2. **重点关注**：
   - Critical 规则必须100%符合
   - High 规则违规需要有充分理由
   - 代码质量和可维护性

3. **沟通方式**：
   - 具体问题给出具体行号和代码
   - 提供明确的改进建议
   - 保持建设性和客观

4. **审查记录**：
   - 保留完整的审查记录
   - 在 PR 中标注审查结果
   - 追踪问题的修复情况
